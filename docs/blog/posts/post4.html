UUID v1 vs v4 vs v7: Which Should You Use in 2025?

Published: November 28, 2025

Choosing the right UUID version is more important than ever. Here’s a no-nonsense comparison of the three most relevant versions today.

UUID v1 – Time-Based (This Tool)

  • 60-bit timestamp + node
  • Naturally sortable by creation time
  • Excellent for logs, audits, message ordering
  • Privacy-safe when node is randomized (as done here)
  • Best for: databases where insertion order matters, distributed tracing

UUID v4 – Pure Random

  • 122 bits of cryptographically strong randomness
  • No timestamp, no ordering
  • Maximum privacy — zero information leakage
  • Worst database index performance (fragmented inserts)
  • Best for: public APIs, tokens, when privacy is paramount

UUID v7 – Unix Timestamp + Random (New Standard)

  • 48-bit Unix milliseconds + 74-bit random
  • Perfectly sortable like v1
  • No MAC/node exposure
  • Only available in newer libraries (PostgreSQL 17+, etc.)
  • Best for: new systems wanting sortability + privacy

FAQ

Is UUID v1 dead?

No. With randomized nodes, it’s making a comeback — especially for analytics and observability.

Should I migrate to v7?

Yes if you control your stack. Otherwise, randomized v1 is still excellent.

In 2025, UUID v1 isn’t obsolete — it’s evolved. Use it confidently with randomization.
